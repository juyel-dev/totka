🤔 What is TOTKA?

TOTKA is a free, serverless file sharing platform that works by converting any file — images, videos, documents, APKs, anything — into a compact text string. That text string is stored in Google Sheets and can be shared with anyone using just a short Post ID like im00042.

To get the original file back, the recipient enters the Post ID and TOTKA's 18-bit decoding engine reconstructs the exact original bytes in their browser — no server involved, no upload required, zero cost.

⚙️ How It Works

1

Compress

Your file's binary data is first compressed using zlib (deflate level 9) — the same algorithm used in ZIP files — to minimize size before encoding.

2

18-bit Encode

The compressed bytes are read into a bit buffer. Every 18 bits are extracted and mapped to a character — either from an 86-character ASCII pool or a Unicode Private Use Area codepoint. This produces a compact, safe text string.

3

Store in Google Sheets

The encoded text is split into 45,000-character chunks (Google Sheets allows up to 50,000 per cell) and stored across multiple storage spreadsheets. A master index tracks every post's metadata.

4

Share with a Post ID

Each upload gets a unique Post ID (e.g. im00042). Share this 7-character ID anywhere — chat, social media, email.

5

Decode & Download

The recipient enters the Post ID. TOTKA fetches the chunks, runs the 18-bit decode entirely in their browser, decompresses with zlib, and triggers a file download — your original file, bit-for-bit identical.

🔬 The Encoding Engine

Character Pool (86 chars):
ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789০১২৩৪৫৬৭৮৯!#$%&*+-./:;<=>?@^_`{|}~₹

Unicode Private Use Area: U+E000 and above
Chunk size: 18 bits → 2¹⁸ = 262,144 possible values
Values 0–85 → ASCII pool character
Values 86–262,143 → Unicode codepoint (safe, never conflicts with text)

Leftover bits at the end are zero-padded to complete a full 18-bit chunk. On decode, zlib's internal End-of-Stream marker automatically discards the padding.

Why 18 bits? Because 2¹⁸ = 262,144 fits perfectly within Unicode's Private Use Area (U+E000–U+F8FF and beyond), giving us a huge value range while staying safe from display conflicts. The system encodes ~2.25 bits of data per bit of output character, achieving efficient compression while remaining fully text-safe.

✨ Features

🆓 100% Free Forever — No server costs, no hosting fees. Runs on Google's free tier (Apps Script + Sheets).
📁 Any File Type — Images, videos, audio, documents, APKs, ZIPs, executables — if it's a file, TOTKA can encode it.
🔒 No Upload to Servers — Files are encoded entirely in your browser. Only the encoded text string is stored — no binary blobs on any server.
♾️ Unlimited Storage Expansion — Each Google Sheet holds 500,000 rows. Adding more storage takes 30 seconds via the Admin Panel.
🔍 Searchable Feed — Browse posts by tags, sort by newest/oldest/A-Z, full-text search across titles, descriptions, and tags.
👤 User Accounts — Create a profile, manage your uploads, link social media, and optionally connect your own personal Google Sheet.
🔞 NSFW Support — Adult content is hidden behind a blur filter by default. Users must explicitly enable NSFW to see 18+ posts.
📱 Mobile-First — Built for phones first. Bottom navigation, infinite scroll, tap-to-reveal, Web Share API support.

⚠️ Limitations

🐢 Large files are slow — Files over 50MB may take several minutes to encode/decode in the browser. The Web Worker helps keep the UI responsive, but encoding is CPU-intensive.
⏱️ GAS response time — Google Apps Script Web Apps have a cold-start latency of 1–3 seconds. The first request after inactivity may be slow.
📏 Google Sheets quota — Free Google accounts have API rate limits. Very high traffic may hit quotas. A Workspace account raises these limits.
🗑️ No chunk deletion — Deleting a post removes it from the feed index, but the encoded chunks in the storage sheet remain to avoid complex row-tracking. Storage is consumed permanently per upload.
